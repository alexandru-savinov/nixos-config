Install Home Assistant declaratively on the rpi5-full NixOS host via a native services.home-assistant wrapper bound to 127.0.0.1:8123 and fronted by a tailscale-serve oneshot, then wire multi-level agent-control tooling (hass-cli + a token-gated HA MCP server) with autonomous, closed verify loops and a single clearly-flagged human onboarding checkpoint.

## Context

**Target host:** `rpi5-full` (aarch64-linux, Raspberry Pi 5, 4GB RAM + 8GB swap + zram), nixpkgs 25.11, stateVersion 25.05. Active host, deploy target `rpi5-full`. The ralphex agent runs ON rpi5 (aarch64-linux), so `systemctl`/`journalctl`/`nc`/`curl`/`sudo cat /run/agenix/...` and `nixos-rebuild build|switch --flake .#rpi5-full` are all valid in-context. HA will be reachable at `https://rpi5.tail4249a9.ts.net:8123`.

**Decision — NATIVE module, not a container.** We use the upstream `services.home-assistant` NixOS module (verified present: `services.home-assistant.config` is a freeform/nullOr YAML option; `services.home-assistant.extraComponents` exists with default `[ "default_config" "met" "esphome" "rpi_power" ]` on aarch64), wrapped by a new `modules/services/home-assistant.nix` exposing `services.home-assistant-tailscale` (options: `enable`, `port` = 8123, `timeZone`, `secretsFile` = nullOr path, `tailscaleServe.{enable,httpsPort}`). The wrapper binds HA to `127.0.0.1` only and fronts it with a `tailscale-serve-home-assistant` oneshot copied from `modules/services/gatus.nix` (which uses a `script=` body with a 60s tailscaled gate, a 60s `nc -z 127.0.0.1 PORT` gate, an idempotent `tailscale serve status | grep -q https:PORT` check before `tailscale serve --bg --https PORT http://127.0.0.1:PORT`, and a port-specific `tailscale serve --bg --https PORT off || true` preStop). `openFirewall` stays `false`.

**CACHE / OOM RULE (critical).** Setting `services.home-assistant.config` is CHEAP — it only renders YAML at `/var/lib/hass/configuration.yaml`, no package rebuild. Overriding `extraComponents` / `extraPackages` / `package` / `customComponents` / the python interpreter triggers a LARGE local aarch64 Python closure rebuild that can OOM the 4GB Pi. Therefore: keep `extraComponents` at the module DEFAULT (`[ "default_config" "met" "esphome" ]` + `"rpi_power"` on aarch64 — i.e. do not set it at all) to guarantee a binary-cache hit. Treat any addition as a deliberate, build-first change. Every build-changing task BUILDS first (`nixos-rebuild build`, never blind `switch`). The AUTHORITATIVE cache guard is the non-mutating eval `nix eval --json .#nixosConfigurations.rpi5-full.config.services.home-assistant.extraComponents` (must equal the default); the `/tmp/ha-build.log` grep is a secondary signal only.

**REVERSE-PROXY HARDENING (mandatory).** Behind any reverse proxy HA returns HTTP 400 on every request unless `config.http` sets `use_x_forwarded_for = true` AND `trusted_proxies`. For same-host tailscale serve the proxy source is loopback, so `trusted_proxies = [ "127.0.0.1" "::1" ]` is correct. Full required block:
```nix
config.http = {
  server_host = [ "127.0.0.1" "::1" ];
  server_port = 8123;
  use_x_forwarded_for = true;
  trusted_proxies = [ "127.0.0.1" "::1" ];
};
```
If a 400 still appears end-to-end through Tailscale Serve, add `"100.64.0.0/10"` (Tailscale CGNAT) to `trusted_proxies` — this fallback is wired into the verify/onFailure steps below.

**Recorder tuning (Pi SQLite write load):** `config.recorder = { purge_keep_days = 5; commit_interval = 30; }`.

**Service identity & resources:** HA runs as user `hass`, config/state dir `/var/lib/hass`. `systemd.services.home-assistant.serviceConfig = { MemoryHigh = "768M"; MemoryMax = "1G"; CPUQuota = "150%"; Nice = 7; }`. HA's own `secrets.yaml` (for `!secret` refs) is provided as an agenix-decrypted file symlinked to `/var/lib/hass/secrets.yaml` via a `systemd.tmpfiles` `L+` rule (precedent: `hosts/sancta-claw/openclaw-service.nix` uses `L+` symlinks), keeping plaintext out of the nix store — GATED on `secretsFile != null` so its absence cannot break the build.

**AGENIX MODEL ON THIS HOST (verified).** agenix is implemented as ACTIVATION SCRIPTS (`config.system.activationScripts.{agenix,agenixChown,agenixInstall,agenixNewGeneration}`), NOT a systemd `agenix.service`. Consequences baked into the tasks below: (1) there is NO `agenix.service` to `journalctl -u` — the canonical runtime proof a secret decrypted is `sudo test -f /run/agenix/<name>` (runtime path confirmed as `/run/agenix/<name>`); (2) `nixos-rebuild switch` exiting 0 is itself the proof activation (incl. agenix) succeeded — an unprovisioned `.age` makes `switch` fail outright, which is exactly the chicken-and-egg trap the gating below avoids; (3) the MCP oneshot's `after = [ "network.target" "agenix.service" ]` (kept for parity with `n8n-mcp-claude.nix`) is a no-op ordering hint — real ordering is guaranteed because the oneshot is `wantedBy multi-user.target`, which runs after activation-script secret install.

**Multi-level agent-control goal.** After HA is up, future agents must be able to configure HA on every level:
- **(a) Nix-declarative** — edit `modules/services/home-assistant.nix` → `services.home-assistant.config` (cheap YAML render). Offline validation: `sudo -u hass hass --script check_config -c /var/lib/hass`.
- **(b) REST API** `http://127.0.0.1:8123/api/` (bearer LLAT) — states, services, `POST /api/config/core/check_config`, and automation/scene/script CRUD via `POST/DELETE /api/config/<domain>/config/<id>`.
- **(c) WebSocket API** `ws://127.0.0.1:8123/api/websocket` (bearer LLAT) — registries (area/device/entity/label), dashboards (`lovelace/config/save`), integration setup (`config_entries/flow`).
- **(d) MCP** — recommended **voska/hass-mcp** (community, `uvx`-runnable, stdio child of Claude Code, env `HA_URL`/`HA_TOKEN`), wired into `~/.claude.json` exactly like `modules/services/n8n-mcp-claude.nix`. NOTE: the `uvx hass-mcp` PyPI dist name is resolved over the network only when Claude Code spawns the MCP child — NOT at deploy — so a wrong name fails at MCP-spawn time, never at build; a non-fatal runtime smoke check is included in Task 7. First-party `/api/mcp` is a noted alternative but exposes only the Assist API.
- **(e) CLI** — nixpkgs `home-assistant-cli` (verified attr; pname `homeassistant-cli`, binary `hass-cli`) (env `HASS_SERVER`/`HASS_TOKEN`); offline `hass --script check_config`.

**CHICKEN-AND-EGG (the bootstrap fix).** Creating the HA owner account is a browser-only first-run wizard, and the Long-Lived Access Token (LLAT) is minted in the UI only AFTER onboarding. ralphex cannot do either. So Tasks 1–4 (Phase A) run fully autonomously with token-OPTIONAL `null` defaults and declare NO `age.secrets` for the token — the build stays green and HA serves the onboarding page. Task 5 is the human-only checkpoint. Tasks 6–7 are gated behind the presence of `home-assistant-token.age` in the worktree and only then wire the token in and verify the authenticated surfaces. Re-running ralphex after the human finishes resumes at the first unchecked box.

**WORKTREE VISIBILITY (critical for the manual handoff).** ralphex runs in a `--worktree` checkout, NOT the main repo dir. A `.age` file the human creates must be visible IN THE WORKTREE ralphex uses. Task 5 instructs the human to create/stage it in that worktree (or the agent to fetch it), and Task 6's gate checks the file relative to the worktree CWD.

**Repo conventions honored throughout:** new modules under `modules/services/`; tailscale-serve oneshot copied from `gatus.nix`; secrets via `lib/secrets.nix` `secret`/`ownedSecret` (host already does `inherit (import ../../lib/secrets.nix { inherit self; }) secret ownedSecret;` and `self` is in scope), registered in `secrets/secrets.nix` as `"<name>.age".publicKeys = users ++ [ rpi5 ];`, referenced via `config.age.secrets.<name>.path`; the `.age` file must EXIST before deploy or agenix activation fails the whole switch; MCP modules target `users = [ "nixos" ]`; never commit to main (ralphex `--worktree`); always `nix fmt` before commit (CI runs `nix fmt -- --check .`); verify-first protocol (hypothesis → minimal non-mutating test → apply → re-verify), never `2>/dev/null` to hide errors.

## Constraints

- **NEVER commit to main.** Use the ralphex `--worktree` isolation; commit on the feature branch via `/commit` only (it runs `nix fmt` first). Verify with `pwd` and `git branch --show-current` before any git operation.
- **Always `nix fmt` before committing.** CI runs `nix fmt -- --check .` and fails on unformatted files. Use `/commit`, never plain `git commit`.
- **NEVER add to `extraComponents` / `extraPackages` / `package` / `customComponents` or override the python interpreter without a deliberate build-first check.** These trigger a large local aarch64 Python rebuild that can OOM the 4GB Pi. Keep `extraComponents` at the module DEFAULT (`[ "default_config" "met" "esphome" "rpi_power" ]`) to guarantee a binary-cache hit. The AUTHORITATIVE guard is `nix eval --json .#nixosConfigurations.rpi5-full.config.services.home-assistant.extraComponents`. Any addition is a separate, build-first change.
- **Do NOT run `nixos-rebuild switch` if the `extraComponents` eval has diverged from the default** (HA/Python would compile from source rather than substitute, risking earlyoom on the 4GB Pi). The `/tmp/ha-build.log` grep is only a secondary signal; the eval is the source of truth. If it diverged, STOP and revert.
- **NEVER inline secrets into the nix store.** HA's `secrets.yaml` is provided only via the agenix-decrypted file symlinked with `systemd.tmpfiles "L+"`; the LLAT lives only in `secrets/home-assistant-token.age` and is read at runtime from `/run/agenix/home-assistant-token`. Never hardcode tokens or paste plaintext into `.nix` files.
- **Bind `127.0.0.1` only.** HA's `http.server_host = [ "127.0.0.1" "::1" ]`; access is exclusively via the Tailscale Serve HTTPS front. `openFirewall` stays `false`; never add a `networking.firewall.allowedTCPPorts` entry for 8123.
- **The `.age` file must EXIST in the worktree before any host config references it.** Declaring `age.secrets.home-assistant-token` (Task 6) before `secrets/home-assistant-token.age` exists makes agenix activation FAIL the whole `switch`. Tasks 1–4 declare NO token `age.secrets`; the gated Tasks 6–7 are guarded by `test -f secrets/home-assistant-token.age` (worktree-relative) AND `sudo test -f /run/agenix/home-assistant-token`.
- **There is NO `agenix.service` on this host.** agenix runs as activation scripts. Never verify secret decryption with `journalctl -u agenix`; use `sudo test -f /run/agenix/<name>` and treat a successful `nixos-rebuild switch` (exit 0) as proof activation/decryption succeeded.
- **The HA MCP `uvx hass-mcp` dist name is resolved at MCP-spawn time over the network, not at deploy.** A wrong name does NOT fail the build/deploy; it only yields a dead MCP server. Surface it via the non-fatal `uvx hass-mcp --help` smoke in Task 7, not by failing a task.
- **Do NOT perform the manual onboarding (human-only).** ralphex cannot drive the browser first-run wizard or mint the LLAT. At Task 5, if `secrets/home-assistant-token.age` is absent in the worktree, STOP and report the pending checkpoint; do NOT fabricate a token, do NOT wire `age.secrets`, do NOT check Task 5/6/7 boxes.
- **Follow verify-first on every fix:** state ONE hypothesis → run the smallest non-mutating test (`nix eval`, `agenix -d`, a single `curl`, `nix-instantiate --parse`, `sudo test -f`) → apply the minimal change → re-run the SAME test. Do not fix multiple things at once. Never suppress errors with `2>/dev/null`.
- **`imports` must be top-level**, never inside `lib.mkIf` (NixOS module-system evaluates imports before conditional logic). Use `lib.mkIf` for conditional `config`, `lib.mkEnableOption` for boolean enables, and `config.age.secrets.<name>.path` (never hardcoded `/run/agenix/...` in `.nix`).
