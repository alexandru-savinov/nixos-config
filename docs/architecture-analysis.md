## How it fits together

## What this repo is

A single NixOS flake driving a small personal fleet of **6 host configurations across 2 architectures**, all stitched together by Tailscale and agenix-encrypted secrets. The real-world roles split cleanly:

- **rpi5-full** (aarch64, the *only* truly active deploy target per CLAUDE.md) — the home server. It imports `rpi5/configuration.nix` wholesale and layers on the home-services stack: n8n workflow automation, Gatus monitoring, NixFrame photo frame, Home Assistant, and the backup-pull job. `rpi5` itself exists almost solely to build the SD image — it is labelled "minimal" but already pulls UniFi-MCP, n8n-MCP/skills, claude-shared and copilot, so the only genuine delta is rpi5-full's 6 extra service imports.
- **sancta-choir** (x86 Hetzner) — Open-WebUI AI chat gateway on the OpenRouter backend, fronted by Tailscale Serve HTTPS.
- **sancta-claw** (x86 CX33) — the dedicated OpenClaw/Kuzea Telegram agent. By far the most complex host (9 sub-files: the systemd unit, self-service watchers, Tailscale-serve front, backup-user + backup-acl, restore + smoke-test DR scripts). All model calls route through a local ZDR sidecar proxy on 127.0.0.1:5780.
- **hermes-claw** (x86 CX33) — Nous Research Hermes agent in a Podman container via an upstream flake module.
- **zero-kuzea** (x86 CX22) — minimal NullClaw Telegram bot (Anthropic Claude directly).

## How config flows

The layering is consistent and worth drawing as lanes: **flake inputs → host builders → host configs → service modules → system fragments → secrets → ops/CI**.

`flake.nix` declares ~15 inputs. Most `follows nixpkgs`; the lone exception is `nixos-raspberrypi`, which deliberately carries its own pinned nixpkgs (25.11) — the entire repo was bumped 25.05→25.11 *to align with it* and silence Home-Manager version warnings (#264). That alignment is load-bearing and fragile.

Each host is built by `nixpkgs.lib.nixosSystem` (or the raspberrypi builder for the two Pis), gets `pkgs-unstable` via specialArgs, and pulls in `agenix`, `home-manager`, `disko` (VPS only) and the hand-rolled inline `agenixModule` (just to add the agenix CLI). Every host imports `hosts/common.nix` (SSH keepalives, zram, fail2ban, flakes, `/bin/bash` symlink, core CLI). Service wrappers under `modules/services/` follow one strong pattern: **bind to 127.0.0.1, expose via a per-service `*-tailscale-serve` oneshot, pull secrets through `age.secrets`.** System fragments under `modules/system/` (nix-ld, ssh-hardened, dev-tools, the sancta-choir-only host/networking) and `modules/users/root.nix` round out the base.

## Secrets and trust

agenix runs **two trust models by design**: the core hosts (rpi5/rpi5-full, sancta-choir) use their SSH host key as the recipient (exposed to the documented fail-open-on-host-key-rotation trap), while the VPS hosts use a standalone `/root/.age/recovery.key` so a DR re-image decrypts on first boot. A subtle coupling: **zero-kuzea is not a distinct recipient at all** — it decrypts its two secrets only because it shares sancta-claw's recovery key, so one key compromise spans both boxes.

## Notable runtime topology

The fleet is mostly star-shaped around Tailscale, with two real cross-host edges: rpi5-full's `backup-pull` rsyncs `/var/lib/openclaw` **from** sancta-claw (rrsync-restricted `backup-user`), and sancta-claw's `restore.sh` pushes the inverse (ssh+restic restore from rpi5). NixFrame is unusual — it has **no listener of its own**; its entire upload ingress is the n8n `nixframe-ui` webhook, making n8n a hard runtime dependency of the photo-frame path. Gatus monitors n8n, Home Assistant and (across hosts) sancta-claw's OpenClaw healthz.

## The dominant theme: four agent runners, two Claude modules

The single biggest structural observation is **product fragmentation**: the same "Telegram + agent + OpenRouter/Anthropic" product is implemented three times across three dedicated VPSes (openclaw-host npm, nullclaw zig, hermes container) — plus a fully-built-but-**dead** fourth implementation, `modules/services/openclaw.nix` (~820 lines, zero importers, confirmed by grep). On top of that, Claude Code itself is provisioned two ways (legacy package-only `claude.nix` on the choir/claw VPSes vs the full `claude-shared` stack on the Pis), pulling the CC binary from two different inputs — a version-skew risk the repo's own header comments flag as a deprecation path.


## Key runtime flows

- **Open-WebUI chat (sancta-choir)** — `user -> tailscale serve :443 -> open-webui :8080 -> openrouter.ai (+ tavily web search); WEBUI_SECRET_KEY/OPENAI_API_KEY/TAVILY_API_KEY from agenix`
- **OpenClaw agent with ZDR enforcement (sancta-claw)** — `user -> Telegram Bot API -> openclaw-host gateway :18789 -> openclaw-zdr-proxy 127.0.0.1:5780 (injects provider.zdr=true, rejects non-ZDR) -> openrouter.ai/api/v1`
- **Cross-host backup + restore loop** — `rpi5-full backup-pull -> ssh rrsync -> sancta-claw backup-user reads /var/lib/openclaw -> tmpfs -> restic encrypted repo; inverse: sancta-claw restore.sh -> ssh rpi5 -> restic restore -> rsync back`
- **NixFrame photo upload (no native listener)** — `user -> tailscale serve :5678 -> n8n /webhook/nixframe-ui -> n8n (in nixframe group) writes photoDir + .trigger -> nixframe slideshow on HDMI/VT7`
- **n8n MCP into Claude Code (rpi5-full)** — `Claude Code -> ~/.claude.json MCP entry -> npx n8n-mcp (stdio) -> N8N_API_URL 127.0.0.1:5678 with N8N_API_KEY from agenix`
- **NullClaw bot (zero-kuzea)** — `user -> Telegram Bot API -> nullclaw gateway :18790 -> api.anthropic.com (claude-sonnet); decrypts via shared sancta-claw recovery key`
